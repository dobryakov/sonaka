# Чек‑лист: Юнит‑тесты для требований (на русском)

Назначение: Проверка качества требований (полнота, ясность, согласованность, измеримость, покрытие) для фичи `001-customer-orders-dashboard`.
Создано: 2025-11-03
Аудитория: Автор спецификации
Глубина: Стандартная

## Полнота требований
- [ ] CHK001 Полностью ли описаны правила включения/исключения заказов для всех статусов, включая частичную оплату/исполнение, отмены и возвраты? [Completeness, Spec §FR-010, Spec §FR-013, Edge Cases]
- [ ] CHK002 Определены ли требования к пустым состояниям для дашборда, списка ранее купленных и деталей заказа? [Completeness, Spec §FR-008, Edge Cases]
- [ ] CHK003 Явно ли перечислены обязательные поля элемента в списке заказов (идентификатор, дата, статус, итоговая сумма)? [Completeness, Spec §FR-003]
- [ ] CHK004 Полон ли состав данных в деталях заказа для каждой позиции (изображение/плейсхолдер, название, количество, цена на момент покупки) и для итогов заказа (промежуточные суммы, общий итог)? [Completeness, Spec §FR-005]
- [ ] CHK005 Определены ли границы агрегирования «ранее купленных» (горизонт данных, какие заказы включаются, группировка по товару vs SKU)? [Completeness, Spec §FR-006, Spec §FR-007, Spec §FR-011, Assumptions]
- [ ] CHK006 Достаточно ли требований к пагинации/бесконечной ленте для очень больших историй (пороги, триггеры, индикаторы загрузки)? [Completeness, Spec §FR-009, Edge Cases]

## Ясность требований
- [ ] CHK007 Недвусмысленно ли определена «Общая сумма заказанного» (что входит/не входит: налоги/доставка исключены) и правила конвертации валют? [Clarity, Spec §FR-002]
- [ ] CHK008 Точно ли заданы критерии «полностью завершённые/оплаченные» с привязкой к конкретным статусам/состояниям? [Clarity, Spec §FR-010]
- [ ] CHK009 Явно ли и проверяемо ли правило исключения частично возвращённых заказов (исключать весь заказ)? [Clarity, Spec §FR-013]
- [ ] CHK010 Однозначно ли сформулирована агрегирующая логика по товарам (варианты/SKU свёртываются в один товар), включая крайние случаи вариантов? [Clarity, Spec §FR-011]
- [ ] CHK011 Чётко ли описаны правила отображения валюты и округления (банковское до 2 знаков) и их универсальность? [Clarity, Spec §FR-012]

## Согласованность требований
- [ ] CHK012 Согласованы ли между собой правила включения/исключения в дашборд‑итогах, списке заказов и списке «ранее купленных» без противоречий? [Consistency, Spec §FR-002, Spec §FR-007, Spec §FR-010, Spec §FR-013]
- [ ] CHK013 Единообразны ли требования к пустым состояниям на всех экранах (условия, принципы сообщений)? [Consistency, Spec §FR-008, Edge Cases]
- [ ] CHK014 Последовательны ли правила валюты/округления между списком заказов, деталями и итогами? [Consistency, Spec §FR-012]

## Качество критериев приёмки (измеримость)
- [ ] CHK015 Имеют ли функциональные требования объективные критерии проверки, особенно для агрегации и фильтрации заказов? [Acceptance Criteria, Spec §FR-002, Spec §FR-007, Spec §FR-010, Spec §FR-013]
- [ ] CHK016 Явно ли измеряется точность итогов и «Общей суммы заказанного» (100% соответствие расчёту)? [Measurability, Spec §SC-003]
- [ ] CHK017 Измеримы ли показатели быстродействия для ключевых путей (загрузка дашборда < 2 с; поиск/открытие заказа < 30 с)? [Measurability, Spec §SC-001, Spec §SC-002]
- [ ] CHK018 Определён ли способ измерения использования списка «ранее купленных» в соответствии с целевым показателем? [Measurability, Spec §SC-004]

## Покрытие сценариев
- [ ] CHK019 Полностью ли описаны основные сценарии: просмотр дашборда, списка «ранее купленных» и деталей заказа, с ожидаемыми результатами? [Coverage, Spec §User Story 1, Spec §User Story 2, Spec §User Story 3]
- [ ] CHK020 Описаны ли альтернативные сценарии (заказы без изображений, мультивалюта, варианты товара)? [Coverage, Spec §FR-005, Spec §FR-012, Spec §FR-011, Edge Cases]
- [ ] CHK021 Заданы ли исключительные сценарии (отменённые/возвращённые/частично оплаченные) с явными правилами обработки? [Coverage, Spec §FR-010, Spec §FR-013]
- [ ] CHK022 Рассмотрены ли сценарии восстановления/деградации при ошибках загрузки данных (подход к ретраям, пользовательские сообщения)? [Coverage, Gap, Assumptions]

## Граничные случаи
- [ ] CHK023 Определены ли нулевые состояния для всех экранов (сообщения, макет)? [Edge Case Coverage, Spec §FR-008, Edge Cases]
- [ ] CHK024 Учтены ли очень большие истории заказов (поведение ленивой загрузки, пороги, UX‑подсказки)? [Edge Case Coverage, Spec §FR-009, Edge Cases]
- [ ] CHK025 Задано ли поведение при отсутствии изображений товаров (плейсхолдер, доступный текст)? [Edge Case Coverage, Spec §FR-005, Edge Cases]
- [ ] CHK026 Чётко ли описано, что при изменении цены/названия после покупки в деталях показываются значения на момент заказа? [Edge Case Coverage, Edge Cases]

## Нефункциональные требования (НФТ)
- [ ] CHK027 Чётко ли привязаны требования по производительности к функциональным областям и условиям (холодный/тёплый кэш, мобильный/десктоп)? [Non-Functional, Gap, Spec §SC-001]
- [ ] CHK028 Определены ли требования по доступности (a11y) для списков, бесконечной ленты, плейсхолдеров и итогов (фокус, читабельность, альтернативный текст)? [Non-Functional, Gap]
- [ ] CHK029 Заданы ли целевые SLO/целевые пороги деградации под нагрузкой и допустимые режимы ухудшения (что упрощается/скрывается)? [Non-Functional, Gap, Performance]

- [ ] CHK031 Описана ли доступность исторических курсов и их корректность, а также поведение при недоступности? [Dependency, Gap, Spec §Assumptions]
